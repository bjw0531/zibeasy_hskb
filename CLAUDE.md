# 프로젝트 가이드라인 (CLAUDE.md)

이 프로젝트에서 작업할 때 아래 규칙을 반드시 따른다.

------------------------------------------------

# 1. 기본 원칙

1. 모든 응답과 설명은 **반드시 한국어로 작성한다.**
2. 모든 UI/UX는 **모바일 환경을 최우선(Mobile First) 기준**으로 설계한다. 단, 웹 서비스 특성상 태블릿과 PC에서도 사용성이 떨어지지 않도록 반응형으로 구현한다. 필요한 경우 데스크톱에서는 레이아웃과 정보 배치를 별도로 최적화할 수 있다.
3. 모든 코드에는 **한글 주석을 반드시 포함한다.**
4. 작업을 시작하기 전에 반드시 아래 절차를 따른다.

작업 절차

1️⃣ 사용자 요청 분석  
2️⃣ 작업 계획(Plan) 작성  
3️⃣ 사용자 승인 요청  
4️⃣ 승인 후 작업 진행  

승인 없이 바로 코드 수정하지 않는다.

------------------------------------------------

# 2. 코드 수정 규칙

코드를 수정할 때는 반드시 다음 절차를 따른다.

1️⃣ 수정할 파일을 먼저 열어 구조를 파악한다.  
2️⃣ 변경 범위를 먼저 설명한다.  
3️⃣ 코드 수정은 **최소 변경(diff 최소화)** 원칙을 따른다.  
4️⃣ 수정한 파일은 닫지 말고 **열어 둔 상태로 유지한다.**  
5️⃣ 수정된 부분이 **명확히 표시되도록 설명한다.**  
6️⃣ 언제든 **이전 상태로 되돌릴 수 있도록 수정 전 상태를 유지한다.**

------------------------------------------------

# 3. UX / 디자인 원칙

이 프로젝트는 **사용자 경험 중심 서비스**이다.

UI를 만들 때 아래 기준을 따른다.

1️⃣ 사용자가 처음 접했을 때 **직관적으로 이해 가능해야 한다.**  
2️⃣ 사용자가 **생각하지 않아도 사용할 수 있는 UX**를 지향한다.  
3️⃣ 모바일 **한 손 사용성**을 고려한다.  
4️⃣ 서비스에 **Wow 포인트가 있는 디자인**을 고려한다.  
5️⃣ 복잡한 UI보다 **단순하고 직관적인 UI**를 우선한다.

스타일링 일관성 규칙

1️⃣ 페이지마다 다른 카드 문법을 만들지 말고, 흰 배경 + 얇은 테두리 + 은은한 그림자 + 비슷한 라운드 값을 기본 체계로 유지한다.  
2️⃣ 간격은 8px, 12px, 16px, 24px 같은 고정 리듬을 우선 사용한다. 답답하면 장식을 늘리기보다 간격부터 정리한다.  
3️⃣ 작은 액션 카드에는 아이콘 → 제목 → 설명 순서를 유지하고, 제목은 어둡고 선명하게 유지한다.  
4️⃣ 강조색은 제목 전체보다 아이콘, 배지, 작은 포인트에 우선 사용한다.  
5️⃣ 특정 섹션만 튀는 그라데이션, 글로우, 특수 질감은 지양하고 기존 카드 스타일을 먼저 재사용한다.

------------------------------------------------

# 4. 금지 사항

다음 행동은 절대 하지 않는다.

❌ 내가 요청하지 않은 라이브러리 설치  
❌ 기존에 잘 작동하는 레거시 코드 수정  
❌ 전체 코드 구조를 임의로 변경  
❌ 승인 없이 코드 수정

------------------------------------------------

# 5. 프로젝트 기술 기준

가능하면 다음 기술을 사용한다.

- TailwindCSS
- OOP 설계
- 디자인 패턴 적용

목표는 **유지보수가 쉬운 구조**이다.

------------------------------------------------

# 6. 파일 구조 원칙

파일이 지나치게 길어질 경우 다음을 고려한다.

1️⃣ 기능 단위 파일 분리  
2️⃣ CSS 분리 가능 여부 검토  
3️⃣ 유지보수성을 고려한 구조 개선  

단, 파일 분리 전에는 **사용자에게 먼저 제안하고 승인받는다.**

------------------------------------------------

# 7. 실행 환경 규칙

Bash에서 Python을 실행할 때는 python3 명령어를 사용한다.

------------------------------------------------

# 8. Git 작업 규칙

이 프로젝트는 git으로 관리한다.

1️⃣ 작업을 실제로 수행해 코드나 문서를 수정했다면 **작업 단위마다 commit을 만든다.**  
2️⃣ 여러 작업을 한 commit에 섞지 말고, **방금 끝낸 요청 범위만 commit** 한다.  
3️⃣ 사용자가 **commit하지 말라고 명시한 경우에만** commit을 생략한다.  
4️⃣ 저장소가 초기화되지 않았다면 먼저 git 저장소 상태를 확인하고 필요한 초기 설정을 진행한다.

------------------------------------------------

# 9. 더 좋은 방법 발견 시

사용자 요청보다 더 나은 방법이 있을 경우

1️⃣ 대안을 먼저 설명한다.  
2️⃣ 장단점을 설명한다.  
3️⃣ 사용자 승인 후 진행한다.

무조건 사용자 요청을 그대로 구현하지 않는다.

----------------------------------------------------

UI/UX 작업은 ui-guide 스킬의 절차를 따른다.
모든 코드 작업은 safe-dev 스킬의 원칙을 따른다.
새 기능 개발·버그 수정은 feature-dev 스킬을 따른다.
UX 분석이 필요하면 ux-review 스킬을 사용한다.


## 4. 작업 프로젝트 참고사항
- 가능하면 Tailwindcss를 사용해.
- 디자인 시에는 frontend-design 스킬을 사용해서 만들어.
- 한 파일이 길어질 경우에는 요청을 완료하기 전에 파일을 분리하여 유지보수가 쉽게 만드는 것을 고려하고 사용자에게 되물어서 양해를 구해.
- css파일은 경우에 따라 분리해서 만드는 것을 권장한다.
- 'python' 명령어를 Bash에서 사용해야 할 때는 'python3'를 사용한다.
- 프로그래밍 디자인 패턴, OOP의 원칙을 적극 채용하여 유지보수가 쉬운 프로그램 제작을 1순위 목적으로 한다.
- 사용자의 요청보다 더 좋은 방식이 있을 때에는 요청을 무작정 수용하기 보다는 더 나은 방식을 제안한다.
- 실제 파일 수정이 있었다면 작업 단위마다 git commit을 남긴다. 사용자가 명시적으로 원하지 않을 때만 예외로 한다.
